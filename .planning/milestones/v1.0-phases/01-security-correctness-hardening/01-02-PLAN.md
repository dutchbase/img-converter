<feature>
  <name>processImage security and correctness fixes</name>
  <files>types/index.ts, lib/imageProcessor.ts, __tests__/imageProcessor.test.ts</files>
  <behavior>
    REQ-101 — Decompression bomb guard:
    - `processImage(buffer, options)` where buffer is a >25 MP image throws an error (does not silently process)
    - The limitInputPixels is set as a Sharp constructor option: `sharp(buffer, { limitInputPixels: 25_000_000 })`
    - A separate `sharp(buffer).metadata()` pre-check is run first; if `(width ?? 0) * (height ?? 0) > 25_000_000`, throw with a recognizable message
    - NOTE: `processImage` throws rather than returning an HTTP response — the route handler converts the throw to a 422. This keeps HTTP concerns out of imageProcessor.

    REQ-103 — ICC preservation:
    - When `options.removeMetadata = true`, the output buffer retains the ICC profile (call `keepIccProfile()`)
    - When `options.removeMetadata = false`, the output buffer retains all metadata (call `withMetadata()`)
    - NEVER call `withMetadata({ exif: {} })` — this pattern is deprecated and silently drops ICC profiles

    REQ-105 — AVIF effort cap:
    - The AVIF case calls `image.avif({ quality, effort: 4 })` — NOT `speed: 6` (speed is not a valid Sharp 0.34.x option)
    - effort: 4 is the Sharp default; making it explicit ensures it is enforced and documented

    REQ-107 — withoutEnlargement from allowUpscaling:
    - When `options.allowUpscaling` is false or undefined and resize is requested, `withoutEnlargement: true`
    - When `options.allowUpscaling` is true and resize is requested, `withoutEnlargement: false`
    - types/index.ts: `ConvertOptions.allowUpscaling?: boolean` added (optional field, undefined = no upscaling)
  </behavior>
